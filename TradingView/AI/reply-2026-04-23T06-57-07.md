# ML Bot Architecture: Timeframe & Entry Strategy

There's no single "best" approach — it depends on your **alpha decay speed**, **latency tolerance**, and **risk profile**. Here's the decision framework:

---

## The Core Tradeoff

| Dimension | Close-Bar Entry | Intrabar Entry |
|-----------|----------------|----------------|
| **Signal quality** | Higher (complete information) | Lower (noisy, partial data) |
| **Execution price** | Worse (slippage, missed moves) | Better (catch wicks, sweep liquidity) |
| **ML accuracy** | Higher (stable features) | Lower (features change mid-bar) |
| **False positive rate** | Lower | Higher (whipsaws trigger fakeouts) |
| **Computational cost** | Low | High (monitor every tick) |
| **Overfitting risk** | Lower | Higher (curve-fits to noise) |

---

## Decision Matrix: Which Approach for Your Situation?

### Choose Close-Bar If:
- You're trading **1H+ timeframes** (daily/weekly holds)
- Your model uses **lagging indicators** (EMA crossovers, momentum)
- You have **limited compute** (can't monitor tick data)
- You're **risk-averse** (avoid false entries)
- Your **alpha has long decay** (position holds hours/days)

### Choose Intrabar If:
- You're trading **5m-15m timeframes** (scalping, high turnover)
- Your edge is **reaction speed** (arbitrage, liquidity sweeps)
- You have **reliable low-latency execution** (<100ms)
- Your model uses **price-action features** (order flow, pivots, FVGs)
- You can tolerate **higher trade frequency** with smaller size

---

## The Hybrid Approach (Recommended for Most)

After building dozens of ML trading systems, the **"Structure + Execution"** split wins:

```
┌─────────────────────────────────────────────┐
│  HIGHER TF (30m-1H): ML Prediction Engine   │
│  ├─ Directional bias (long/short/neutral)   │
│  ├─ Key levels (pivots, OBs, Fibs)          │
│  └─ Position sizing (volatility-adjusted)   │
└──────────────────┬──────────────────────────┘
                   │ context (not entry)
                   ▼
┌─────────────────────────────────────────────┐
│  LOWER TF (1m-5m): Execution Engine         │
│  ├─ Price-action triggers (sweep/reclaim)   │
│  ├─ Volume/momentum confirmation            │
│  └─ Stop-loss placement (below sweep low)   │
└─────────────────────────────────────────────┘
```

**Why this works:**
1. ML runs on **stable features** (30m/1H) → better generalization
2. Entry uses **reactive logic** (1m/5m) → better fills
3. You validate ML prediction **before** committing capital
4. Risk is defined by **structure** (sweep low), not arbitrary stops

## Feature Engineering by Entry Type

### Close-Bar Model (Stable)
```python
features = {
    # Price-based (normalized)
    'rsi_14': ta.rsi(close, 14),
    'ema_ratio': ema_9 / ema_21,
    'atr_pct': atr(14) / close,
    
    # Volume-based
    'volume_vs_ma': volume / sma(volume, 20),
    'obv_slope': slope(obv, 10),
    
    # Structure-based
    'distance_to_pivot': (close - pivot_r1) / atr(14),
    'adx': adx(14),  # trend strength
    
    # Time-based
    'hour_of_day': hour,
    'day_of_week': dayofweek,
}
```

### Intrabar Model (Reactive)
```python
features = {
    # Current bar dynamics (change every tick!)
    'wick_ratio': (high - low) / close,
    'close_position': (close - low) / (high - low),  # 0=low, 1=high
    'delta': close - open,
    
    # Microstructure
    'bid_ask_pressure': (close - low) / (high - low) - 0.5,
    'volume_intensity': volume / time_since_bar_open,
    
    # Context from higher TF
    'htf_bias': ml_30m_prediction,  # cached from last close
    'distance_to_htf_level': abs(close - htf_pivot) / close,
}
```

**Critical:** Don't mix unstable features (intrabar) with stable training (close-bar). Your model will learn patterns that don't exist in live trading.

## The Overfitting Trap (Your Previous Issue)

**Why intrabar fails for most ML beginners:**
- Training on **simulated intrabar data** is unreliable (you don't know the tick sequence)
- Models learn **spurious correlations** between wick patterns and outcomes
- Live execution differs from backtest (spread, slippage, latency)

**The fix:**
1. **Train on close-bar data** (clean, available, reproducible)
2. **Deploy with intrabar execution** (manual rules, not ML)
3. Validate with **walk-forward on tick data** (separate from training)
