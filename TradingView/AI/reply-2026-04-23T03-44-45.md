## 1. Timeframe Selection

**Recommendation: 15m or 30m primary, with 4H/1D context**

| Timeframe | Trades/Day | Signal Quality | Why |
|-----------|-----------|----------------|-----|
| **5m** | 5-15 | Low | Too noisy, spread eats edge, SHAP loses interpretability |
| **15m** ✅ | 2-5 | Medium | Sweet spot for crypto 24/7, enough bars for patterns |
| **30m** ✅ | 1-3 | High | Best for your goal (1-3 trades/day), cleaner signals |
| **1H** | 0-2 | High | Too few setups for active trading |

**Your Problem:** 5m generates ~288 bars/day. With 300 features, you're overfitting to noise. ML can't distinguish signal from microstructure randomness at that granularity.

**Fix:** Move to **30m primary** with **4H trend filter**. This gives you:
- ~48 bars/day (manageable)
- Cleaner technical patterns
- SHAP values become interpretable again

## 2. Do You Need to Feed Trading Strategy to ML?

**Yes — but as "soft labels" or constraints, not hard rules.**

### Option A: Hybrid Approach (Recommended)
Use your strategy as a **filter**, then ML for **entry timing**:

```
1. Strategy generates "trade candidates" (e.g., EMA crossover + RSI < 40)
2. ML model scores each candidate (probability of success)
3. Only take trades where ML score > threshold
```

### Option B: Label Engineering
Instead of raw price returns, label your data with **strategy-aligned outcomes**:

| Traditional Label | Better Label |
|-------------------|--------------|
| "Price up 1% in next 10 bars" | "Hit 2R target before 1R stop" |
| Binary up/down | Return / Max adverse excursion |

**Your Problem:** Raw price prediction is noisy. Frame it as "will this setup reach target before stop?"

## 4. Why Your LightGBM + SHAP Failed (Diagnosis)

| Symptom | Root Cause | Fix |
|---------|------------|-----|
| Bad backtest | Overfitting to 5m noise | Move to 30m, reduce features to 20-30 |
| SHAP values confusing | Too many correlated features | Remove collinear features (RSI + Stoch + Williams %R = similar) |
| Poor generalization | Look-ahead bias or data leakage | Use **purged k-fold CV** with embargo periods |
| No edge in live | Training on wrong objective | Optimize for risk-adjusted returns, not accuracy |

### Critical Fix: Purged Cross-Validation
Standard CV leaks future info into training. For time series:

```
Fold 1: Train [0-1000], Test [1000-1100], Embargo [1000-1020] ← 20-bar gap
Fold 2: Train [0-1100], Test [1100-1200], Embargo [1100-1120]
...
```

This prevents "peeking" and gives realistic backtests.

## 5. Recommended Pipeline

```
1. Data Collection
   └── 30m bars for BTC (or TAO) + 4H/1D context
   └── 2 years of history minimum

2. Feature Engineering
   └── 20-30 curated features (see Tier 1-4 above)
   └── Remove features with >0.9 correlation

3. Label Creation
   └── "Did price reach 2R target before 1R stop within 20 bars?"
   └── Binary: 1 = hit target first, 0 = hit stop first

4. Train/Validation Split
   └── Purged CV with embargo
   └── LightGBM with early stopping

5. SHAP Analysis
   └── Global: Which features matter most?
   └── Local: Why did model predict THIS trade?

6. Strategy Integration
   └── Only trade when:
       ├── ML probability > 0.65
       ├── 4H trend aligns with trade direction
       ├── ADX > 20 (avoid chop)
       └── Position sizing based on Kelly or fixed 1% risk
```
